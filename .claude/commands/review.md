Проведи ревью всех открытых, ещё не смёрженных PR в репозитории. Работай в три фазы.

## Фаза 1: Подготовка

1. Получи список открытых PR: `gh pr list --state open --json number,title,headRefName`.
2. Если открытых PR нет — сообщи об этом и заверши работу.
3. Прочитай `structure.md` (правила документации), `templates/city.md` и `templates/walking-route.md` (шаблоны).

## Фаза 2: Ревью каждого PR

Для каждого PR выполни:

1. **Получи изменения**: `gh pr diff <номер>`.
2. **Прочитай описание PR**: `gh pr view <номер>`.
3. **Прочитай полные файлы**, затронутые в PR (не только diff), чтобы проверить контекст.

### Что проверять

#### Соответствие structure.md
- Статьи о городах соответствуют шаблону `templates/city.md`: обязательные разделы присутствуют, порядок разделов правильный.
- Пешеходные маршруты соответствуют шаблону `templates/walking-route.md`: обязательные разделы присутствуют, порядок разделов правильный.
- Язык: всё на русском, кроме устоявшихся английских названий.

#### Консистентность навигации
- Если добавлен новый файл — проверь, что обновлены: `structure.md`, `mkdocs.yml`, `README.md`, `travel-knowledge-base/index.md`, и `index.md` соответствующей директории (если есть).
- Если файл удалён или переименован — проверь, что ссылки на него убраны/обновлены везде.

#### Содержание и здравый смысл
- Информация правдоподобна и не содержит очевидных фактических ошибок.
- Нет персональных данных (GDPR-compliance).
- Нет пустых разделов без пометки «нет данных».
- Таблицы корректно отформатированы (заголовки, разделители, выравнивание).
- Нет дублирования информации внутри статьи.

#### Форматирование
- Уровни заголовков: H1 — название, H2 — разделы, H3 — подразделы.
- Единообразное форматирование списков и ссылок.
- Нет лишних пустых строк, битых ссылок или артефактов.

## Фаза 3: Вердикт

Для каждого PR прими одно из двух решений:

### Если всё корректно
- Одобри PR: `gh pr review <номер> --approve --body "<краткий комментарий>"`.

### Если есть замечания
- Оставь ревью с замечаниями: `gh pr review <номер> --request-changes --body "<список замечаний>"`.
- В теле ревью перечисли все найденные проблемы, сгруппированные по категориям выше.
- Для каждой проблемы укажи файл и конкретное место.

## Важно

- **НИКОГДА не мёржи PR.**
- **НИКОГДА не пуш коммиты в чужие ветки.**
- Если PR содержит только изменения в конфигурации (CI, mkdocs.yml и т.п.) — проверяй синтаксис и логику, но не применяй правила structure.md.
